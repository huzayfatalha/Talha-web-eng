1.1 INTRODUCTION

Background:
In the modern business environment, managing customer relationships effectively
is crucial for organizational success. Sales teams and small businesses often
struggle with organizing and tracking potential customers (leads) efficiently.
Traditional methods of lead management rely on scattered spreadsheets,
paper-based systems, or expensive enterprise software that is not suitable for
small organizations. These outdated approaches result in:

• Data Fragmentation: Lead information scattered across multiple files and
  systems
• Time Wastage: Significant time spent searching for customer details
• Data Duplication: Multiple copies of same customer information with
  inconsistencies
• Lost Opportunities: Inability to quickly access customer information leads to
  missed follow-ups
• Poor Decision Making: Lack of organized data prevents effective business
  analysis

Problem Statement:
Small businesses and sales teams require a simple, accessible, and cost-effective
solution to centralize lead information, enable quick searching and filtering,
and maintain up-to-date customer records. The existing solutions are either too
expensive (enterprise CRM systems like Salesforce) or too basic (spreadsheets).
There is a need for a beginner-friendly web application that bridges this gap—
providing essential CRM functionality without complexity or high cost.

---

1.2 MOTIVATION

The motivation for developing Lead Tracking AI stems from several key observations:

1. Market Gap: Most CRM solutions target large enterprises, leaving small
   businesses underserved
2. Accessibility: Web-based applications are more accessible than desktop
   software or paper systems
3. Educational Value: This project demonstrates full-stack web development using
   fundamental technologies (PHP, MySQL, HTML, CSS, JavaScript)
4. Practical Utility: The system can be immediately used by real businesses for
   genuine lead management
5. Scalability: Simple architecture allows for future enhancements without
   complete redesign
6. Open Source: Uses free technologies that anyone can access and modify

1.3 OBJECTIVES

The primary objectives of this project are:

1. Develop a functional lead management system that allows users to create,

3. Create an intuitive user interface that is easy to use for non-technical users

5. Demonstrate full-stack web development using PHP, MySQL, HTML, CSS, and

7. Deploy on local XAMPP server demonstrating complete deployment workflow

2.1 REQUIREMENT ANALYSIS & DESIGN SPECIFICATION

2.1.1 Overview

Lead Tracking AI is designed as a three-tier web application architecture
consisting of:
1. Presentation Layer (Frontend): HTML, CSS, JavaScript
2. Business Logic Layer (Backend): PHP
3. Data Layer (Database): MySQL

This separation ensures scalability, maintainability, and clear separation of
concerns. The system is designed to work on XAMPP (Apache + MySQL) for local
deployment.

2.1.2 Functional Requirements

User Management:
• Users must be able to create new accounts with username and password
• Users must be able to login with valid credentials
• Users must be able to logout and end their session
• Users must see welcome message showing their username after login
• Passwords must be stored (currently plain text for educational purposes)
• Usernames must be unique in the system
• Session must persist across page navigation

Lead Management (CRUD Operations):
• Users must be able to add new leads with: Name, Email, Phone, Date Added
• Users must be able to view all leads in a table format
• Users must be able to edit lead information (Name, Email, Phone)
• Users must be able to delete leads with confirmation dialog
• Date Added field must be read-only (cannot be edited)
• Users must see success/error messages for all operations

Search and Filtering:
• Users must be able to search leads by Name or Email
• Search must be case-insensitive
• Search results must show matching leads only
• Search must display count of results found
• Users must be able to clear search and see all leads again

Pagination:
• System must display 5 leads per page
• Pagination controls must show: Previous, Page Numbers, Next
• Current page must be highlighted
• Disabled buttons must show for first page (Previous) and last page (Next)
• Page indicator must show "Page X of Y"

Data Validation:
• Name field: Required, minimum 2 characters
• Email field: Required, must be valid email format (contain @)
• Phone field: Optional, must contain only numbers and symbols (+, -, (), space)
• All fields must be trimmed of whitespace
• Empty forms must not submit

Security Requirements:
• All pages except login/signup require authentication
• Direct URL access without login must redirect to login page
• Session timeout after inactivity
• Input sanitization using mysqli_real_escape_string
• Prevention of duplicate usernames

2.1.3 Non-Functional Requirements

Performance:
• Page load time must be less than 2 seconds
• Search results must display within 1 second
• Database queries must be optimized for speed

Usability:
• Interface must be intuitive for non-technical users
• All buttons must be clearly labeled
• Error messages must be clear and helpful
• Success messages must confirm actions taken

Availability:
• System must be accessible 24/7 when server is running
• No downtime for features

Maintainability:
• Code must follow standard practices with comments
• Database schema must be normalized
• Files organized in logical folders structure

SQL Queries Used:

CREATE DATABASE IF NOT EXISTS crm_project;
USE crm_project;

CREATE TABLE customers (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) NOT NULL,
    phone VARCHAR(20),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE admins (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL
);

INSERT INTO admins (username, password) VALUES ('admin', '12345');

The database was successfully created in phpMyAdmin with proper normalization

3.1.3 Backend Implementation

Database Connection (db.php):
• Used mysqli_connect() for database connection
• Implemented error handling for connection failures
• Connection reused across all pages for efficiency

Authentication System (login.php & signup.php):
• Implemented user registration with validation
• Username: 3-20 characters, letters/numbers/underscores only
• Password: Minimum 4 characters
• Duplicate username prevention
• Session management for logged-in users
• Password validation on login

Lead Management System:

ADD LEAD (add_customer.php):
• Form validation (name length, email format, phone format)
• Server-side validation using PHP
• Insert query: INSERT INTO customers (name, email, phone) VALUES (...)
• Auto-redirect to View Leads after successful addition
• Success message display

VIEW LEADS (view_customers.php):
• Query: SELECT * FROM customers ORDER BY id DESC
• Pagination logic: 5 leads per page
• Offset calculation: (page - 1) * 5
• Count total leads for pagination control
• Display in HTML table format

SEARCH FUNCTIONALITY (view_customers.php):
• Search query: SELECT * FROM customers WHERE name LIKE '%search%'
  OR email LIKE '%search%'
• Case-insensitive search
• Display result count
• Clear search functionality

EDIT LEAD (edit_customer.php):
• Fetch lead by ID: SELECT * FROM customers WHERE id = 'id'
• Pre-fill form with current data
• Update query: UPDATE customers SET name='...', email='...', phone='...'
  WHERE id='...'
• Disable Date Added field (read-only)
• Validation on update

DELETE LEAD (view_customers.php):
• Delete query: DELETE FROM customers WHERE id = 'id'
• Confirmation dialog before deletion
• Refresh page after deletion

3.1.5 Key Technical Decisions

Decision 1: No Framework Used
Rationale: To demonstrate fundamental web development concepts and understand
how things work at the core level, without abstraction of frameworks.

Decision 2: Procedural PHP (Not OOP)
Rationale: Simpler for beginners to understand, faster development, appropriate
for project scope.

Decision 3: Plain Text Password Storage
Rationale: Educational project focused on functionality. Production systems
would use bcrypt hashing.

Decision 4: Local XAMPP Deployment
Rationale: Portable, easy to setup, good for learning and demonstrations.

Decision 5: 5 Leads Per Page
Rationale: Compromise between showing enough data and maintaining page
performance and readability.

5.2 LIMITATIONS

Technical Limitations:

1. Scalability Limit
   Current: System handles up to ~1000 leads efficiently
   Future: Large databases (10,000+ leads) may require database indexing
   Impact: Currently suitable for small businesses; may need optimization
           for enterprise use

2. Password Storage
   Current: Passwords stored in plain text
   Limitation: Not secure for production use
   Recommendation: Use bcrypt hashing in production version
   Impact: Educational system demonstrates concept; production needs security

3. No Email Integration
   Current: System doesn't send email confirmations or reminders
   Limitation: All communication is within system only
   Future: Email notifications would enhance functionality
   Impact: Users must manually check system for updates

4. Single User Type
   Current: All users have same permissions (admin-level access)
   Limitation: No role-based access control
   Future: Admin, manager, user roles could be implemented
   Impact: Not suitable for large teams with different access needs

5. No Mobile Interface
   Current: Desktop/tablet web only
   Limitation: No native mobile app
   Future: Mobile app or responsive mobile design needed
   Impact: Users must access from computer

6. Limited Reporting
   Current: No built-in reports or analytics
   Limitation: Users must analyze data manually
   Future: Dashboard with charts and reports needed
   Impact: Limited business intelligence capabilities

7. No Data Export
   Current: Cannot export leads to CSV or PDF
   Limitation: Data locked within system
   Future: Export functionality would increase utility
   Impact: Difficult to integrate with other systems

8. Session Timeout
   Current: No automatic session timeout
   Limitation: Session continues indefinitely if page left open
   Future: Implement timeout and re-authentication
   Impact: Potential security risk on shared computers

Operational Limitations:

1. Deployment Complexity
   Current: Requires XAMPP and database setup by user
   Limitation: Not as simple as cloud SaaS deployment
   Impact: Technical knowledge required for setup

2. Limited Documentation
   Current: Basic README and code comments
   Limitation: Could use more detailed tutorials
   Impact: Steep learning curve for some users

3. No Customer Support
   Current: Community-based support only
   Limitation: No dedicated support team
   Impact: Issues may take time to resolve

Project Scope Limitations:

1. Small Business Focus
   Current: Designed for small teams (< 10 people)
   Limitation: May not meet enterprise needs
   Impact: Large organizations need different system

2. English Language Only
   Current: All interface text in English
   Limitation: Not accessible to non-English users
   Future: Multi-language support needed
   Impact: Limited to English-speaking market

3. No Advanced Analytics
   Current: Basic search and view only
   Limitation: No predictive analytics or AI features
   Future: ML integration for lead scoring possible
   Impact: Competitive CRM systems offer more analysis
